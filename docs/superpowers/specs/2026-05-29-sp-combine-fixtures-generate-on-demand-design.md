# Generate `sp_combine` parity fixtures on demand

- **Date:** 2026-05-29
- **Status:** Approved (design)
- **Branch:** `claude/bold-johnson-132a38`

## Problem

Two strudel-parity test fixtures are committed to the source tree:

| File | Size | Lines |
| --- | --- | --- |
| `crates/modular_core/src/pattern_system/__fixtures__/sp_combine_chain2.json` | 18.3 MB | 1,284,801 |
| `crates/modular_core/src/pattern_system/__fixtures__/sp_combine.json` | 1.75 MB | 125,755 |

They are **deterministic** — pure functions of strudel `@strudel/mini` 1.2.5 (pinned) and a fixed case grid in `scripts/gen-sp-fixture.mjs` (regenerating produces byte-identical output, confirmed). They are effectively a committed binary artifact: not human-reviewable, exactly reconstructable in **2.7 s**.

Costs:

- **Working tree** — 20 MB bloats every clone/checkout.
- **History growth** — already committed twice on this branch; the husky pre-commit auto-regenerates them when the generator changes, so every future combiner/grammar change appends another ~18 MB blob to history, unbounded.
- **Diff noise** — a 1.28 M-line file is unreviewable; any regen shows as a massive diff.

Single consumer: `crates/modular_core/tests/sp_combine_strudel_parity.rs`, which reads them at **test runtime** via `fs::read_to_string` (not `include_str!`), so the fixtures need not exist at compile time — only when the test runs.

## Goals

- Stop committing the two big `sp_combine` fixtures; generate them on demand.
- Keep full strudel-parity coverage (no reduction of the cross-product).
- `cargo test` still works standalone (self-healing), with no silent loss of coverage.
- Purge the already-committed blobs from this branch's history.
- Run the Rust DSP/parity tests in CI (currently they run nowhere automated).

## Non-goals

- `peggy_parser_parity.json` (15 KB, 822 lines, generated by a TS test) stays committed — too small to matter, different generation path.
- `grammar.generated.ts` handling is unchanged.
- No reduction of parity coverage, no Git LFS, no compaction of the JSON shape.
- Full-workspace `cargo test` in CI (which would require the napi build + audio deps) is out of scope; CI runs `cargo test -p modular_core`, where the parity tests live.

## Decisions (user-approved)

1. **Direction:** generate on demand, stop committing.
2. **Missing-fixture behavior:** the Rust test auto-generates (shells out to node) when a fixture is absent; panics with an actionable message if generation fails. No silent skip.
3. **History:** purge the existing blobs from branch history (rewrite + force-push).
4. **CI:** add the Rust tests to CI (new addition to the approved design).

## Design

### 1. Untrack + ignore

- Add to `.gitignore`:
  ```
  crates/modular_core/src/pattern_system/__fixtures__/sp_combine.json
  crates/modular_core/src/pattern_system/__fixtures__/sp_combine_chain2.json
  ```
- No separate `git rm` needed — the history purge (§5) removes the paths from every commit, so the final tree never tracks them.

### 2. Rust test self-heals — `tests/sp_combine_strudel_parity.rs`

- Add an `ensure_fixtures()` helper, called once before the fixtures are read.
- Logic:
  - Fixture dir = `env!("CARGO_MANIFEST_DIR") / src/pattern_system/__fixtures__`.
  - Repo root = `CARGO_MANIFEST_DIR / ../..`.
  - If **either** `sp_combine.json` or `sp_combine_chain2.json` is missing, run `Command::new("node").arg(repo_root/"scripts/gen-sp-fixture.mjs")` with `current_dir(repo_root)`. One invocation writes both fixtures.
  - Guard with a `std::sync::OnceLock` (or `Once`) so parallel test threads generate at most once per test binary.
  - On node-not-found / non-zero exit / still-missing after run → `panic!` with: `sp_combine fixtures missing and generation failed — run \`yarn install\` then \`yarn gen:sp-fixtures\`. Cause: <stderr/err>`.
- When the fixtures are present on disk (locally generated once, or pre-generated by the yarn/CI step), the helper is a cheap existence check — zero generation overhead. First run after a fresh clone costs +2.7 s once.
- Both existing read sites (`fixture_path("sp_combine.json")`, `fixture_path("sp_combine_chain2.json")`) call `ensure_fixtures()` first.

### 3. Husky pre-commit

- Remove the block that regenerates and `git add`s the two fixtures when `scripts/gen-sp-fixture.mjs` is staged (a `git add` of a now-ignored path would error). The fixtures regenerate at test time instead.
- Keep the `grammar.generated.ts` regeneration block unchanged.

### 4. yarn wiring

- `package.json`: change `test:rust` from `cargo test` to `yarn gen:sp-fixtures && cargo test`. This guarantees fixtures exist on the yarn path (parallel-safe, +2.7 s once); bare `cargo test` falls back to the §2 self-heal.

### 5. Purge branch history

- The fixtures were introduced in `cefb8c1` (`sp_combine.json`) and `2071258` (`sp_combine_chain2.json`), and modified in `f1e44a1`. All are branch-only commits (after the merge-base with master), so rewriting them does not touch master's history.
- **Primary tool:** `git filter-repo --path <sp_combine.json> --path <sp_combine_chain2.json> --invert-paths` (install: `brew install git-filter-repo`). Strips both paths from every commit that contains them; downstream branch commits get new SHAs.
- **Safety rails:** back up the branch ref first (`git branch backup/pre-purge`); the repo is a git worktree, so prefer running the rewrite from the primary checkout or a fresh clone to avoid surprising the shared object store; pass `--force` as filter-repo requires.
- **Fallback** if filter-repo is unavailable or the worktree blocks it: recreate the branch (cherry-pick the commits with the fixture hunks excluded) or `git filter-branch --index-filter 'git rm --cached --ignore-unmatch <paths>'`.
- After the rewrite, add **one functional commit** carrying §1–§4 + §6 (`.gitignore`, the Rust self-heal, the husky edit, the `test:rust` change, the new `ci.yml`).
- **Force-push** the branch. Flag: any collaborator on this branch must reset to the rewritten ref.

### 6. CI — new `.github/workflows/ci.yml`

`release.yml` triggers only on `v*` tags, so it would never run the parity tests pre-merge. Add a dedicated CI workflow:

- **Triggers:** `pull_request` and `push` (branches: `master`).
- **Runner:** `ubuntu-latest`.
- **Steps:**
  1. `actions/checkout@v4` with `submodules: recursive` (the Rust build needs the vendored submodules).
  2. `actions/setup-node@v4` (node 24, `cache: yarn`).
  3. `dtolnay/rust-toolchain@stable`.
  4. `yarn install --frozen-lockfile` (provides `@strudel/*` for the generator).
  5. `yarn test:unit` (vitest — currently only run at release; now gains per-PR coverage).
  6. `yarn gen:sp-fixtures` then `cargo test -p modular_core` — run directly as the CI step (not via `yarn test:rust`). Scoping to `modular_core` keeps the parity coverage while avoiding the napi `modular` crate, which would otherwise pull in `yarn build` (native module) and the Linux audio deps (`libasound2-dev`, `libjack-jackd2-dev`, `libudev-dev`).
- The full-workspace `yarn test:rust` (`yarn gen:sp-fixtures && cargo test`) remains the local and release path; only CI is scoped to `-p modular_core`. If full-workspace rust tests in CI are wanted later, add the native build + audio deps (mirroring `release.yml`) as a follow-up.

## Implementation sequence

1. Branch backup ref.
2. Apply §2 (Rust self-heal), §1 (`.gitignore`), §3 (husky), §4 (`test:rust`), §6 (`ci.yml`) — stage but do not yet rely on the old fixtures being gone.
3. Verify §Verification locally (with fixtures deleted) before rewriting history.
4. Run §5 history purge (filter-repo), then create the single functional commit from step 2.
5. Force-push; confirm CI green on the PR.

## Verification

- `git ls-files | grep sp_combine` → empty.
- `git log --all --oneline -- <both fixtures>` → empty (after purge).
- `rm -f` both fixtures, then `cargo test -p modular_core` → self-heal regenerates (2.7 s) and parity tests pass.
- `yarn test:rust` passes from a clean tree.
- Determinism: `yarn gen:sp-fixtures` twice → byte-identical (already confirmed).
- Report `.git` pack size and working-tree size before/after.
- CI: open/update the PR, confirm the new `ci.yml` job runs unit + `modular_core` rust tests and is green.

## Risks & mitigations

| Risk | Mitigation |
| --- | --- |
| Parallel cargo test threads race to generate | `OnceLock`/`Once` guard around generation. |
| Pure-Rust dev without `yarn install` | Self-heal panics with an actionable `run yarn install` message. |
| History rewrite is destructive / worktree-shared object store | Backup ref; run from primary checkout or fresh clone; `--force`; documented force-push. |
| Force-push disrupts collaborators | Flag in PR; collaborators reset to rewritten branch. |
| CI rust tests dragging in napi/audio deps | Scope CI to `cargo test -p modular_core`. |
| node missing on a runner | CI sets up node before the rust step; self-heal requires node by design. |
